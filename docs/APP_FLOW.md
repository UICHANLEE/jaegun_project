# 목양실 10:12 앱 흐름도

![앱 흐름도](flow.svg)

## 사용자 흐름

```mermaid
flowchart LR
  A[참가자 접속] --> B[밝은 레크레이션 대기 화면]
  C[관리자 접속] --> D[관리자 콘솔]
  D --> E{전환 버튼}
  E -->|크라임씬으로 전환| F[/api/mode = crime/]
  E -->|되돌리기| G[/api/mode = recreation/]
  F --> H[25대 휴대폰 폴링]
  G --> H
  H --> I{현재 모드}
  I -->|recreation| B
  I -->|crime| J[크라임씬 힌트 화면]
  J --> O{조 배정 완료?}
  O -->|예| P[후면 카메라로 증거 촬영]
  P --> Q[사진 압축 및 메모 작성]
  Q --> R[/api/evidence 조별 저장/]
  R --> S[같은 조 증거 갤러리]
  O -->|아니오| J
  J --> K[힌트코드 입력]
  K --> L[힌트 해제]
  L --> M{12개 모두 열림?}
  M -->|아니오| J
  M -->|예| N[최종 추리 봉투]
```

## 25대 동시 전환 설계

- 참가자 페이지는 `/api/mode`를 0.85~1.35초 랜덤 간격으로 확인합니다.
- 랜덤 지터가 있어 25대 휴대폰이 같은 순간에 요청을 몰아치지 않습니다.
- Vercel 배포 시 `UPSTASH_REDIS_REST_URL`, `UPSTASH_REDIS_REST_TOKEN`을 설정하면 모든 서버리스 인스턴스가 같은 상태를 읽습니다.
- 관리자는 버튼 클릭 한 번으로 Redis의 `crime-scene:mode` 값을 갱신합니다.
- 참가자 기기는 다음 폴링 때 `recreation` 또는 `crime` 상태를 받아 화면을 바꿉니다.

## 조별 증거 갤러리

- 참가자 이름이 관리자 조 편성표에 등록된 경우에만 `/api/evidence`를 사용할 수 있습니다.
- 촬영 사진은 휴대폰에서 긴 변 720px JPEG로 압축되며, 게임 상태 폴링과 분리되어 전환 속도에 영향을 주지 않습니다.
- Redis에는 조별 목록을 분리해 저장하고 각 조의 최근 사진 18장만 유지합니다.
- 갤러리는 열 때만 사진을 요청하며, 다른 조의 이름으로는 해당 조 증거를 읽을 수 없습니다.
- 관리자는 조별 수사 현황에서 각 조의 힌트 획득 상태와 증거사진을 확인합니다. 힌트 상태와 사진 개수는 가볍게 자동 확인하고, 사진이 바뀔 때만 원본 목록을 다시 불러옵니다.
- `전체 초기화`는 모드, 타이머, 조 편성, 입장 명단, 힌트 기록과 증거사진을 비우고 새 게임 ID를 발급합니다. 참가자 휴대폰은 새 ID를 감지하면 이름과 로컬 힌트 진행도까지 초기화합니다.

## 운영 화면

| 화면 | 역할 | 주요 액션 |
| --- | --- | --- |
| 참가자 `/`, `index.html` | 팀별 휴대폰 화면 | 레크레이션 대기, 크라임씬 진입, 힌트코드 입력, 조별 증거 촬영·열람 |
| 관리자 `jaegunadmin.html` | 진행자가 사용하는 콘솔 | 모드 전환, 힌트코드 테스트, 정답 확인 |
| 참가자 `player.html` | 참가자 모바일 페이지 백업 경로 | 기본 링크와 동일한 참가자 화면 |
| 미리보기 `preview.html` | 리허설/디자인 확인 | 관리자와 참가자 화면을 한 화면에서 비교 |

## 완성도 개선 후보

- 관리자 비밀번호 또는 비공개 경로 추가
- 전환 3초 카운트다운 연출
- 참가자 팀명 입력 및 팀별 진행률 수집
- 힌트 오픈 로그 저장
- 최종 정답 제출 폼 추가
- Vercel KV/Upstash 연결 후 현장 리허설
